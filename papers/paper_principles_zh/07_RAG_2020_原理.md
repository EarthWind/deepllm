# RAG 原理导读

**论文标题**：`Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks`
**年份**：`2020`
**主题**：`检索增强生成`
**定位**：把参数化知识与外部检索系统结合起来的主线起点之一。
**论文链接**：[arXiv](https://arxiv.org/abs/2005.11401)

## 1. 代表公式 / 关键表达
- 经典表达：`p(y | x) = Σ_z p_η(z | x) p_θ(y | x, z)`，其中 `z` 是召回文档。
- 这里把知识来源拆成检索分布 `p_η` 和生成分布 `p_θ` 两部分。

## 2. 这篇论文要解决什么问题
- 纯参数化模型必须把知识都压进参数里，更新昂贵，且容易过时。
- 知识密集型任务需要更强的事实召回能力，仅靠模型记忆不可靠。
- 作者要解决的问题是：如何在生成时接入外部文档，使模型既能读文档又能生成答案。

## 3. 核心思想
- 把检索器和生成器组合起来，让模型先从语料库找相关文档，再条件生成答案。
- 显式区分 parametric memory 与 non-parametric memory。
- 在训练或推理阶段利用检索结果增强知识覆盖和事实性。

## 4. 方法原理拆解
- 检索器根据输入查询召回若干候选文档，生成器把问题和文档共同编码后输出答案。
- 文档可以被看作外部上下文，动态提供模型参数外的知识来源。
- 这种架构让“知识更新”可以通过更新索引或文档库来完成，而不必重训整个模型。

## 5. 代码实现结构
- RAG 的实现不是单一模型类，而是一套 `检索系统 + 生成模型 + 数据编排` 的组合系统。
- 如果按代码模块拆，通常可以分成 `Document Store`、`Retriever`、`Reranker`、`Prompt Builder / Fusion Layer`、`Generator` 五部分。
- 因此 RAG 的“代码结构”比 BERT、GPT 更接近一个服务系统而不是单个神经网络。

### 5.1 文档库与索引层
- 原始文档通常先被切分成 chunk，再建立向量索引或其他检索索引。
- 代码上常见流程是：
  `load docs -> split -> embed -> build index`
- 这一层决定了模型能不能找到正确知识，因此是 RAG 系统的底座。

### 5.2 Retriever
- Retriever 负责根据输入问题召回 top-k 候选文档。
- 在现代工程实现里，它往往会独立成一个类或服务，例如：
  `retriever.retrieve(query, top_k=5)`
- 如果是可训练 RAG，retriever 也可能参与训练；如果是工程型 RAG，常直接使用独立 embedding 模型。

### 5.3 Reranker 与证据筛选
- 召回后的文档未必都好，因此很多实现会再做一层 rerank。
- reranker 可以是交叉编码器，也可以是规则排序器。
- 在代码结构里，这一步的职责是“从召回可用，提升到证据可靠”。

### 5.4 Generator
- Generator 是最终负责输出答案的模型。
- 最简单的做法是把 `query + retrieved docs` 拼接成 prompt，然后交给生成模型。
- 如果是原论文式实现，检索结果和生成器的结合会更紧密；如果是应用实现，常直接采用 prompt 拼接接口。

### 5.5 一个最小 RAG 骨架

```python
class RAGPipeline:
    def __init__(self, retriever, generator, reranker=None):
        self.retriever = retriever
        self.generator = generator
        self.reranker = reranker

    def answer(self, query):
        docs = self.retriever.retrieve(query, top_k=8)
        if self.reranker is not None:
            docs = self.reranker.rank(query, docs)[:4]
        prompt = build_prompt(query, docs)
        return self.generator.generate(prompt), docs
```

### 5.6 如果做训练版 RAG
- 可训练 RAG 往往需要同时考虑 `retriever loss` 与 `generator loss`。
- 代码结构上可能会把检索分布 `p_η(z|x)` 和生成分布 `p_θ(y|x,z)` 一起纳入 forward。
- 这比普通 prompt-based RAG 更学术，也更难训练和调试。

### 5.7 实现时最值得注意的点
- 文档切分、embedding 模型、召回 top-k、上下文拼接格式，常常比“换个更强 LLM”更影响效果。
- 证据放进上下文不代表模型一定会正确引用，所以 prompt 模板与引用格式也很关键。
- 如果你在工程里做 RAG，代码设计最好把 `索引层`、`召回层`、`生成层` 解耦，否则后续很难替换组件。

## 6. 训练或推理范式
- RAG 类方法的重点不仅在模型训练，还在检索语料、索引和召回质量。
- 生成器必须学会利用召回文档，而不是无视它们继续幻觉式生成。
- 训练时也常面临检索误召回、文档噪声和证据融合问题。

## 7. 为什么它有效
- 外部文档为模型提供可追溯证据，显著减轻纯参数记忆的负担。
- 知识密集型任务往往需要最新、长尾或细粒度事实，检索系统更适合承担这部分职责。
- 生成器擅长组织自然语言答案，检索器擅长找事实，两者互补。

## 8. 局限与争议
- 如果检索器召回失败，生成器通常也会随之失败。
- RAG 会引入系统复杂度，包括索引维护、chunk 切分、rerank 和上下文预算。
- 把文档喂进去不代表模型一定会正确使用证据，证据利用仍是难点。

## 9. 对后续研究的影响
- 奠定了企业知识库问答、文档助手、私有化问答系统的核心思想。
- 推动检索、embedding、rerank、agentic RAG 等完整应用栈发展。
- 使“模型参数不是全部知识载体”成为行业共识。

## 10. 你应该怎样读这篇
- 读这篇时要把注意力放在系统分工，而不仅是模型结构。
- 如果你做应用，建议把这篇和 ReAct 一起看，理解知识获取与工具调用的关系。
- RAG 是工程问题与模型问题强耦合的典型案例。
