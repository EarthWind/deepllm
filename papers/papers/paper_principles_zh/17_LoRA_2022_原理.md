# LoRA 原理导读

**论文标题**：`LoRA: Low-Rank Adaptation of Large Language Models`
**年份**：`2021/2022`
**主题**：`参数高效微调`
**定位**：PEFT 生态最具代表性的论文之一，改变了微调大模型的门槛。
**论文链接**：[arXiv](https://arxiv.org/abs/2106.09685)

## 1. 代表公式 / 关键表达
- 核心表达：`W' = W + BA`，其中 `rank(BA) = r` 且 `r` 很小。
- 它假设任务适配所需的权重更新位于低秩子空间。

## 2. 这篇论文要解决什么问题
- 全量微调大模型成本极高，需要大量显存、存储和训练时间。
- 很多应用只需要轻量定制，但不值得为每个任务保存一整份模型副本。
- 作者要解决的是如何用极少可训练参数完成有效适配。

## 3. 核心思想
- 冻结原模型权重，只在部分线性层上学习低秩增量矩阵。
- 把大的权重更新分解为两个较小矩阵相乘，从而显著减少训练参数量。
- 推理时可将增量并入原权重，部署成本低。

## 4. 方法原理拆解
- 假设任务适配所需的权重更新本质上位于较低秩子空间中。
- 因此无需学习完整矩阵，只需学习一个低秩近似更新。
- 常见做法是在 attention 的投影矩阵上挂载 LoRA 模块，以较小代价获得较强适配能力。

## 5. 代码实现结构
- LoRA 的实现方式非常适合作为代码插桩模块理解：不是重写整个模型，而是在某些线性层旁边挂一个低秩分支。
- 因此代码结构通常拆成 `基础线性层`、`LoRA A/B 矩阵`、`冻结主干参数`、`推理合并逻辑` 四部分。
- 从工程上看，LoRA 最优雅的一点在于它几乎不改变原模型前向接口。

### 5.1 LoRA 注入点
- 最常见的注入位置是 attention 里的 `q_proj`、`k_proj`、`v_proj`、`o_proj` 等线性层。
- 有些实现也会把 LoRA 挂到 FFN 层，但注意力投影层是最经典的主战场。
- 代码层面你通常只需要把原来的 `nn.Linear` 换成一个包装版模块。

### 5.2 LoRA 线性层骨架
- 原始线性层权重 `W` 冻结不动。
- 另外添加两个可训练矩阵 `A` 和 `B`，其乘积构成低秩增量。
- 最终前向等价于：
  `xW + xBA * scaling`

### 5.3 一个最小实现骨架

```python
class LoRALinear(nn.Module):
    def __init__(self, in_features, out_features, rank, alpha):
        super().__init__()
        self.base = nn.Linear(in_features, out_features, bias=False)
        self.base.weight.requires_grad = False
        self.lora_A = nn.Linear(in_features, rank, bias=False)
        self.lora_B = nn.Linear(rank, out_features, bias=False)
        self.scaling = alpha / rank

    def forward(self, x):
        return self.base(x) + self.lora_B(self.lora_A(x)) * self.scaling
```

### 5.4 训练时怎么工作
- 优化器只需要看到 `lora_A` 和 `lora_B` 的参数。
- 基础模型参数全部冻结，因此显存和优化器状态都大幅下降。
- 从训练脚本角度看，最关键的一步常常只是“过滤出 requires_grad=True 的参数”。

### 5.5 推理时怎么处理
- 一种方式是保留分支，推理时继续走 `base + lora_delta`。
- 另一种方式是把 `BA` 直接合并回原权重，得到一个新的等效矩阵。
- 这也是为什么 LoRA 既适合训练，也适合部署管理。

### 5.6 实现时最值得注意的点
- `rank`、`alpha`、dropout、注入模块范围都会显著影响效果。
- 如果多个 LoRA 适配器要热切换，代码设计上最好把 adapter 单独管理，不要直接覆盖主权重。
- LoRA 的前向很简单，但训练效果高度依赖“挂在哪”和“rank 设多大”。

## 6. 训练或推理范式
- 训练只更新 LoRA 参数，显存和优化器状态开销显著下降。
- 同一个基础模型可以挂接多个不同任务的 LoRA，方便管理和切换。
- 这使个人与小团队也能参与大模型定制。

## 7. 为什么它有效
- 许多任务适配不需要重写模型全部知识，只需在少量方向上调整行为。
- 低秩假设在实践中足够有效，使性能和成本达到很好的平衡。
- 冻结主干还能减少灾难性遗忘和训练不稳定。

## 8. 局限与争议
- 并非所有任务都能在极低秩下达到最佳效果。
- LoRA 参数注入位置、秩大小、目标模块选择都影响结果。
- 它适合适配，但不是降低基础模型推理成本的办法。

## 9. 对后续研究的影响
- 推动 PEFT 成为大模型微调标准范式。
- 为 QLoRA 等后续低成本训练方案奠定基础。
- 极大降低了学术界和工业界的定制门槛。

## 10. 你应该怎样读这篇
- 读这篇时要抓住“适配往往只需低维更新”这个核心假设。
- 如果你做私有化微调，这篇几乎是第一必读文献。
- 和 QLoRA 连读能看懂低成本微调是如何一步步被压低门槛的。
