# DPO 原理导读

**论文标题**：`Direct Preference Optimization: Your Language Model is Secretly a Reward Model`
**年份**：`2023`
**主题**：`偏好优化`
**定位**：后 RLHF 时代最重要的简化方法之一，显著降低偏好训练复杂度。
**论文链接**：[arXiv](https://arxiv.org/abs/2305.18290)

## 1. 代表公式 / 关键表达
- 代表性目标：`L_DPO = -log σ(β log π(y_w|x)/π_ref(y_w|x) - β log π(y_l|x)/π_ref(y_l|x))`。
- 直觉上就是提高优选回答概率、压低拒选回答概率，并受参考模型约束。

## 2. 这篇论文要解决什么问题
- 传统 RLHF 需要 SFT、奖励模型和 PPO，多阶段管线复杂且不稳定。
- 很多团队希望保留偏好优化效果，但不想承担完整 RLHF 的工程负担。
- 作者试图找到一种直接从偏好对训练策略模型的方法。

## 3. 核心思想
- 不显式训练奖励模型，也不跑 PPO。
- 把偏好学习转化为一个对比式监督目标，直接优化策略模型。
- 利用偏好对中“被选回答”和“被拒回答”的相对关系学习。

## 4. 方法原理拆解
- DPO 基于策略和参考模型之间的相对概率比，构造可直接优化的目标函数。
- 本质上是在让模型提高优选答案概率、降低劣选答案概率，同时受参考模型约束。
- 它把原本 RLHF 中隐含的奖励优化过程“折叠”进了一个更简洁的监督式目标中。

## 5. 代码实现结构
- DPO 的实现非常适合作为“损失函数替换”来理解。
- 从代码结构上，通常拆成 `policy model`、`reference model`、`preference pair loader`、`DPO loss` 四部分。
- 与 RLHF 相比，它不需要单独的 reward model 和 PPO trainer，因此训练代码会更像标准监督训练。

### 5.1 两个模型角色
- `policy model` 是当前正在训练的模型。
- `reference model` 是冻结的参考模型，通常来自 SFT checkpoint。
- DPO 的关键就是比较这两个模型对 `chosen / rejected` 两种回答的相对偏好。

### 5.2 偏好数据加载
- 数据集样本通常包含三段：`prompt`、`chosen`、`rejected`。
- 代码上需要分别把 `prompt + chosen` 和 `prompt + rejected` 编码成两路输入。
- 然后计算 policy 与 reference 在这两路输入上的 log probability。

### 5.3 DPO Loss
- 训练的核心就是一个自定义 loss，而不是复杂采样回路。
- 换句话说，整个训练 loop 看起来和普通 SFT 很像，只是 loss 不再是单一路径交叉熵。
- 这也是 DPO 工程上普及很快的原因。

### 5.4 一个最小实现骨架

```python
def dpo_loss(pi_chosen, pi_rejected, ref_chosen, ref_rejected, beta=0.1):
    margin = beta * ((pi_chosen - pi_rejected) - (ref_chosen - ref_rejected))
    return -torch.log(torch.sigmoid(margin)).mean()


for batch in dataloader:
    pi_chosen = policy.log_prob(batch["prompt"], batch["chosen"])
    pi_rejected = policy.log_prob(batch["prompt"], batch["rejected"])
    with torch.no_grad():
        ref_chosen = ref_model.log_prob(batch["prompt"], batch["chosen"])
        ref_rejected = ref_model.log_prob(batch["prompt"], batch["rejected"])
    loss = dpo_loss(pi_chosen, pi_rejected, ref_chosen, ref_rejected)
    loss.backward()
```

### 5.5 为什么工程上更轻
- 不需要 reward model 训练阶段。
- 不需要 PPO rollout、优势估计和在线采样更新。
- 训练接口与现有 SFT 代码兼容度很高，因此很多团队只需增加一个新 loss 函数。

### 5.6 实现时最值得注意的点
- `log_prob` 计算必须和 token mask、padding mask 严格对齐，否则 chosen/rejected 比较会失真。
- `beta` 超参数会影响偏好强度与稳定性，需要调节。
- DPO 虽然简单，但仍然高度依赖高质量偏好对数据。

## 6. 训练或推理范式
- 训练只需要偏好对数据和一个参考模型，整体实现比 RLHF 更简单。
- 稳定性通常更好，也更容易与现有 SFT 流程衔接。
- 因此它迅速成为社区常用偏好优化方法。

## 7. 为什么它有效
- 偏好信息本质上是相对排序，未必一定要通过显式奖励模型中转。
- 约束到参考模型附近可以防止策略过度偏移。
- 直接对比学习减少了 RL 优化中的高方差问题。

## 8. 局限与争议
- 效果依赖偏好数据质量和参考模型选择。
- 它简化了 RLHF 管线，但不自动解决对齐目标定义问题。
- 在某些长程行为优化场景中，显式 RL 仍可能有独特价值。

## 9. 对后续研究的影响
- 极大降低了偏好优化门槛，成为主流后训练工具之一。
- 推动很多团队从 RLHF 迁移到更轻量的 preference optimization。
- 帮助社区把焦点更多放回数据质量与目标设计本身。
