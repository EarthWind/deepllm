# ReAct 原理导读

**论文标题**：`ReAct: Synergizing Reasoning and Acting in Language Models`
**年份**：`2023`
**主题**：`Agent / 推理与行动`
**定位**：agent 领域最经典的“思考 + 行动”交替范式之一。
**论文链接**：[arXiv](https://arxiv.org/abs/2210.03629)

## 1. 代表公式 / 关键表达
- 核心循环可写成：`Thought_t -> Action_t -> Observation_t -> Thought_{t+1}`。
- 它不是单次 `p(y|x)`，而是带外部反馈的多步状态更新过程。

## 2. 这篇论文要解决什么问题
- 纯推理型 prompt 缺少与外部环境交互的能力，纯工具调用又常缺少高层规划。
- 现实任务往往需要一边思考，一边观察外部反馈，再调整下一步。
- 作者要解决的是如何让模型在文本中统一表示思考、动作和观察。

## 3. 核心思想
- 把推理链和动作序列交织在一起，而不是先想完再行动。
- 使用格式化轨迹，例如 Thought、Action、Observation 循环。
- 让模型基于环境反馈持续更新内部计划。

## 4. 方法原理拆解
- Thought 负责局部推理与计划，Action 负责调用工具或环境操作，Observation 提供外部反馈。
- 下一轮 Thought 会基于 Observation 调整策略，形成闭环。
- 这种表示把规划与执行统一进同一条文本轨迹中。

## 5. 代码实现结构
- ReAct 的实现重点不是模型层，而是一个 `agent loop`。
- 代码结构通常拆成 `Prompt Builder`、`Action Parser`、`Tool Executor`、`Loop Controller` 四部分。
- 从工程视角看，ReAct 更像一个运行时框架，把大模型嵌进“思考-行动-观察”的状态机中。

### 5.1 Prompt Builder
- 系统会把当前问题、历史 Thought/Action/Observation 轨迹一起拼成 prompt。
- 因此 prompt 本身承担了“agent memory”的角色。
- 代码上经常就是一个不断 append 轨迹文本的字符串构造器。

### 5.2 Action Parser
- 模型输出后，系统需要区分它给出的是 Thought 还是 Action。
- 一般会要求模型按固定格式输出，例如：
  `Thought: ...`
  `Action: Search[query]`
- 然后用 parser 把 Action 名称和参数拆出来。

### 5.3 Tool Executor
- 如果当前步输出的是 Action，系统就调用外部工具执行。
- 工具返回结果后，封装成 Observation 再拼回对话上下文。
- 这一步本质上把文本 agent 接到了外部环境上。

### 5.4 Loop Controller
- ReAct 不是单次生成，而是多轮循环直到拿到最终答案或达到步数上限。
- 所以代码里通常有一个 while loop 作为核心骨架。
- 每轮循环都做：生成 -> 解析 -> 执行动作 -> 写回 observation。

### 5.5 一个最小实现骨架

```python
def run_react(model, question, tools, max_steps=8):
    trajectory = []
    for _ in range(max_steps):
        prompt = build_react_prompt(question, trajectory)
        output = model.generate(prompt)
        thought, action = parse_react_output(output)
        trajectory.append(("thought", thought))
        if action is None:
            return extract_final_answer(output), trajectory
        obs = tools[action.name](*action.args)
        trajectory.append(("action", action.raw))
        trajectory.append(("observation", obs))
    return None, trajectory
```

### 5.6 实现时最值得注意的点
- ReAct 最难的地方通常不是调用工具，而是输出格式稳定性和循环控制。
- 如果 parser 不稳，agent 很容易在第二三步就崩掉。
- 工程上常会加入最大步数、异常处理、空结果重试和最终答案抽取规则。

## 6. 训练或推理范式
- 论文更多是提示范式与实验验证，而不是复杂训练新算法。
- 但其轨迹格式后来被广泛用于 agent 数据构造、SFT 和评测。
- 它对环境交互任务尤其有价值，比如检索、网页操作、知识问答。

## 7. 为什么它有效
- 很多任务不是静态问答，而是信息逐步揭示的闭环决策问题。
- 把推理显式化有助于规划，把行动显式化有助于接入外部世界。
- 模型在观察反馈后可以纠正初始误判，而不是一次性输出最终答案。

## 8. 局限与争议
- 显式 Thought 轨迹可能冗长，也容易泄露中间过程。
- 轨迹格式脆弱，提示词设计对结果影响明显。
- 真实复杂 agent 还需要更强的记忆、回溯、工具调度和错误恢复机制。

## 9. 对后续研究的影响
- 成为绝大多数 agent 框架的思想原型之一。
- 推动工具调用从单步函数使用演化为多步交互式任务解决。
- 帮助社区建立了“reasoning and acting”一体化视角。

## 10. 你应该怎样读这篇
- 读这篇时要抓住闭环：Thought 不只是解释，还是决定下一步 Action 的状态变量。
- 如果你做 agent，ReAct 基本是绕不开的起点。
- 它和 Toolformer 互补，一个偏何时调工具，一个偏如何在回路中使用工具。
