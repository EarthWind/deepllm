# GPT-3 原理导读

**论文标题**：`Language Models are Few-Shot Learners`
**年份**：`2020`
**主题**：`规模扩展 / 上下文学习`
**定位**：现代大模型时代真正爆发的代表作之一，few-shot 与 in-context learning 由此成为中心议题。
**论文链接**：[arXiv](https://arxiv.org/abs/2005.14165)

## 1. 代表公式 / 关键表达
- 训练目标仍是标准自回归：`p(x) = Π_t p(x_t | x_{<t})`。
- few-shot 时可写成：`p(y | task_description, demonstrations, x)`，体现 in-context learning。

## 2. 这篇论文要解决什么问题
- 大家想知道：当模型继续扩大后，是否可以不用微调，仅靠上下文示例完成任务。
- 更一般地说，模型是否能在 prompt 中“临时学习”任务模式。
- 作者试图用极大规模实验验证语言模型的上下文学习能力边界。

## 3. 核心思想
- 继续沿用 decoder-only 因果语言模型，不在架构上做大改动。
- 主要通过增加参数规模、语料规模和训练计算量，观察能力随规模如何变化。
- 系统对比 zero-shot、one-shot、few-shot 设置，突出上下文学习的现象。

## 4. 方法原理拆解
- few-shot 的本质是把任务描述和示例放入上下文，让模型根据示例模式延续输出。
- 模型参数并没有更新，但上下文中的示例像临时外部工作记忆，塑造了当前行为。
- 这种机制后来被广泛理解为 in-context learning 的经验现象，而非经典意义上的参数学习。

## 5. 代码实现结构
- GPT-3 在代码结构上并没有引入全新架构，本质上仍是更大号的 `decoder-only Transformer`。
- 如果从实现角度拆分，通常仍是 `Embedding 层`、`Decoder Block 堆叠`、`Final Norm`、`LM Head` 四部分。
- 它真正重要的地方在于：同一套简单结构在规模足够大时，开始表现出很强的上下文学习能力。

### 5.1 基础主干结构
- `token embeddings` 和 `position embeddings` 负责把离散 token 序列映射到连续表示。
- 主干由大量 `Masked Self-Attention + FFN` block 堆叠组成。
- 输出经过最终归一化后接 `LM head`，预测每个位置的下一个 token。

### 5.2 因果注意力与上下文窗口
- GPT-3 的实现逻辑仍建立在因果 mask 上，因此模型只能看到左侧内容。
- 但由于上下文窗口更重要，工程上会特别关注长 prompt 的编码、内存占用和推理缓存。
- few-shot 示例、任务说明和输入都只是前缀 token，因此对模型来说它仍然只是单次自回归续写。

### 5.3 推理接口的代码视角
- 所谓 zero-shot / one-shot / few-shot，本质上不是三种不同模型，而是三种不同的输入构造方式。
- 在代码层面可统一成：
  `logits = model(prompt_tokens)`
- 差别仅在 `prompt_tokens` 中是否包含任务描述、1 个示例或多个示例。

### 5.4 一个最小实现骨架

```python
class GPT3LikeModel(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.wte = nn.Embedding(config.vocab_size, config.hidden_size)
        self.wpe = nn.Embedding(config.max_position, config.hidden_size)
        self.blocks = nn.ModuleList([GPTBlock(...) for _ in range(config.num_layers)])
        self.ln_f = nn.LayerNorm(config.hidden_size)
        self.lm_head = nn.Linear(config.hidden_size, config.vocab_size, bias=False)

    def forward(self, input_ids, past_key_values=None):
        pos = torch.arange(input_ids.size(1), device=input_ids.device).unsqueeze(0)
        x = self.wte(input_ids) + self.wpe(pos)
        causal_mask = build_causal_mask(input_ids.size(1), input_ids.device)
        new_past = []
        for i, block in enumerate(self.blocks):
            x, past = block(x, causal_mask, past_key_values[i] if past_key_values else None)
            new_past.append(past)
        x = self.ln_f(x)
        return self.lm_head(x), new_past
```

### 5.5 为什么 KV Cache 在 GPT-3 类实现里尤其重要
- 生成时如果每一步都重算全部历史 token，成本会随输出长度快速上升。
- `past_key_values` 会缓存前面 token 的 `K/V`，新 token 只需和缓存交互。
- 这使长文本生成和交互式推理在工程上更可行，也是今天所有 GPT 类实现的标配思路。

### 5.6 few-shot 是如何体现在程序里的
- 你可以把任务描述、示例输入、示例输出、当前问题全部拼成一个大 prompt。
- 代码上不需要单独写分类头、翻译头、问答头；模型只做统一 next-token prediction。
- 这也是 GPT-3 最震撼人的一点：任务切换往往只是改 prompt，不是改模型类定义。

### 5.7 实现时最值得注意的点
- 当模型规模变大后，真正困难的往往不是 block 定义，而是并行训练、初始化、数值稳定和通信效率。
- 训练代码虽然在逻辑上简单，但工程上需要极强的分布式基础设施。
- 如果自己写 toy 版本，最先要做对的是：因果 mask、权重共享、shift labels、KV cache 接口。

## 6. 训练或推理范式
- 关键不是新结构，而是极端规模化训练。
- 论文覆盖大量 benchmark，展示不同规模下性能曲线与任务敏感性。
- 它让整个领域开始重视 prompt 设计、上下文长度与示例选择。

## 7. 为什么它有效
- 大模型在预训练中学到大量模式压缩能力，提示样例能激活相应行为轨迹。
- 统一语言建模目标让不同任务都能被写成“续写”问题。
- 随着规模增加，模型更能利用少量上下文建立局部任务归纳。

## 8. 局限与争议
- few-shot 能力强但不稳定，提示顺序、措辞和示例质量影响极大。
- 成本很高，且论文时期模型仍存在明显事实性错误和推理脆弱性。
- 能力现象很惊艳，但理论解释仍不充分。

## 9. 对后续研究的影响
- 推动 prompt engineering、上下文学习和 API 式大模型交互成为主流。
- 让很多原本依赖微调的任务开始探索“直接提示是否足够”。
- 是后续 ChatGPT、对齐训练和 agent 接口繁荣的重要前提。
